# Codex × Claude 협업 규칙서

> 최종 수정: 2026-04-13

## 1. 목적

이 프로젝트는 초등 3학년부터 중학교 3학년까지의 7단계 문해력 진단평가 **문항은행**, **판정 로직**, **웹 서비스**를 구축하는 것을 목표로 한다.

Codex와 Claude는 같은 작업 폴더를 공유하지만, 동시에 같은 파일을 수정하지 않는다.
협업의 목표는 속도보다 정합성, 문항 수보다 평가 타당도, 초안 생성보다 검수 가능성을 우선하는 것이다.

## 4. 절대 규칙

1. 같은 파일을 동시에 수정하지 않는다.
2. 파일 소유권이 없는 파일은 읽기만 하고, 수정이 필요하면 담당자에게 넘긴다.
3. 모든 문항은 하나의 `primary_domain`을 가져야 한다.
4. 모든 문항은 하나의 `achievement_standard`와 연결되어야 한다.
5. 지문 ID는 영역 중립형(`P-{단계}-{번호}`)으로 사용한다.
6. 문항이 독립형이면 `passage_id`는 `null`로 둔다.
7. 평가 철학이나 단계 기준이 애매하면 Claude가 임의 확장하지 않고 Codex 기준을 따른다.
8. 문항 수를 채우기 위해 평가 초점을 흐리는 문항은 만들지 않는다.
9. 태도/정의적 요소는 단계 판정 문항에 직접 넣지 않는다.
10. JSON 구조를 바꿀 때는 먼저 Codex 기준 문서(`reference/schema.json`)를 확인한다.

## 6. ID 규칙

### 지문 ID (영역 중립형)
- 형식: `P-{단계}-{번호}`
- 예: `P-1-001`, `P-4-003`, `P-7-002`

### 문항 ID
- 형식: `R-{단계}-{영역약칭}-{번호}`
- 예: `R-1-VOC-001`, `R-5-CRT-004`

### 영역 코드
| 코드 | 영역명 |
|------|--------|
| `VOC` | 어휘력 |
| `LIT` | 사실적 이해 |
| `INF` | 추론적 이해 |
| `CRT` | 비판적 이해 |
| `STR` | 구조 파악 |

---

## 7. 영역 분류 기준

### VOC
- 낱말 뜻, 유의어, 문맥 의미, 표현 의미, 기본 어휘 이해

### LIT
- 중심 내용, 세부 정보, 순서 확인, 내용 간추리기, 요약

### INF
- 내용 예측, 생략 내용 추론, 까닭 추론, 함축 의미 추론, 의도/관점 추론

### CRT
- 사실/의견 구분, 출처 신뢰성, 타당성 판단, 표현 적절성 평가, 자료 근거 평가

### STR
- 처음-가운데-끝, 문단 역할, 글의 짜임, 전개 방식, 순서 배열, 설명/논증 구조

### 경계 처리 원칙
- 중심 내용이면 `LIT`
- 짜임과 구조면 `STR`
- 글에 직접 없는 내용 추론이면 `INF`
- 사실/의견, 신뢰성, 타당성이면 `CRT`

---

## 8. 단계별 생성 원칙

### 1~2단계
- 짧고 친숙한 지문
- 배경지식 의존 최소화
- 발문은 직접적이고 단순하게
- 선택지는 명확하게 구분

### 3~4단계
- 구조 파악과 추론 부담 증가
- 단일 글 기준 타당성/표현 평가 도입
- 인접 단계 간 질적 차이 명확화

### 5~7단계
- 복합 자료, 다양한 글 유형 반영 가능
- 메타인지 성취기준은 간접 측정만 허용
- 근거 충분성, 전제, 관점, 논증 방법 평가 포함
- 시사 주제 사용 가능하되 배경지식 의존 금지

---

## 9. 문항 품질 규칙

1. 정답은 지문 근거로 설명 가능해야 한다.
2. 오답은 매력적이어야 하지만 함정형이어서는 안 된다.
3. 정답만 유독 길거나 짧지 않게 한다.
4. 부정형 발문은 최소화한다.
5. 사실/의견 구분은 참/거짓으로 표현하지 않는다.
6. 출처 신뢰성 문항은 독립형으로도 가능하다.
7. 해설은 학생이 읽고 납득 가능한 수준으로 쓴다.
8. 상위 단계 문항은 전반적으로 더 높은 인지 부담을 가져야 한다.

## 10. 작업 흐름

### Claude 작업 순서
1. 해당 단계의 passages 초안 생성
2. passages 메타데이터 입력
3. items 초안 생성
4. `primary_domain`, `achievement_standard`, `level_differentiation` 입력
5. 지문 참조 무결성 점검
6. **`node scripts/validate.js` 실행하여 자동 검증**
7. 자체 검토 후 Codex에 전달

### Codex 검수 순서
1. 구조 검토
2. 단계 위계 검토
3. 성취기준 연결 검토
4. 영역 분류 검토
5. 문항 품질 검토
6. 수정 지시 또는 승인

## 11. 승인 상태

### review_status 값
| 값 | 의미 | 누가 설정하는가 |
|----|------|----------------|
| `draft` | 초안 | Claude |
| `reviewed` | 구조 검토 완료 | Codex |
| `approved` | 운영 가능 수준 승인 | 사람 |

Claude는 기본적으로 `draft`까지만 올린다.
`reviewed` 전환은 Codex 검수 후 진행한다.
`approved` 전환은 사람 최종 확인 후 진행한다.

---

## 12. 충돌 방지 규칙

- Claude가 `items.json` 수정 중이면 Codex는 해당 파일을 직접 수정하지 않는다.
- Codex가 규칙 문서를 수정 중이면 Claude는 기존 규칙 기준으로 새 파일을 만들지 않는다.
- 규칙 변경이 생기면 먼저 문서 반영 후 생성 작업을 계속한다.
- 대량 수정 전에는 항상 현재 스키마를 다시 확인한다.

## 13. Git 워크플로우

### 브랜치 전략
- `main` — 승인된 안정 버전
- `claude/level-{N}-items` — Claude가 N단계 문항 작업 시 사용
- `codex/review-level-{N}` — Codex가 N단계 검수 시 사용
- `web/feature-{name}` — 웹 서비스 기능 개발

### 작업 흐름
```
Claude: claude/level-3-items 브랜치에서 작업
  → validate.js 통과 확인
  → PR 생성 (main ← claude/level-3-items)

Codex: PR에서 검수
  → 수정 지시 또는 승인
  → 승인 시 review_status 변경 후 머지

사람: 최종 확인 후 approved 전환
```

### 커밋 메시지 규칙
- `feat: 3단계 LIT 문항 5개 추가` — 문항/지문 생성
- `fix: R-3-INF-002 성취기준 수정` — 검수 반영
- `chore: validate.js 영역별 통계 추가` — 도구 개선
- `review: level-2 검수 완료 (reviewed)` — 검수 상태 변경

## 14. 소통 프로토콜

### 검수 요청 (Claude → Codex)
검수 요청 시 다음 정보를 포함한다:

```markdown
## 검수 요청: {단계}단계 {영역/전체}

- 파일:
  - 통합 파일 사용 시: item-bank/items.json, item-bank/passages.json
  - 레벨 분리 파일 사용 시: item-bank/level-{N}/items.json, item-bank/level-{N}/passages.json
- 문항 수: {N}개 (신규 {N}개, 수정 {N}개)
- 지문 수: {N}개
- 자동 검증: 통과/미통과
- 특이사항: {있으면 기재}
```

### 수정 지시 (Codex → Claude)
수정 지시 시 다음 형식을 사용한다:

```markdown
## 수정 지시: {단계}단계

### 우선순위: 높음/보통/낮음

| 문항 ID | 수정 유형 | 수정 내용 |
|---------|----------|----------|
| R-3-CRT-001 | 영역 변경 | CRT → LIT (사실 확인 문항) |
| R-3-INF-002 | 발문 수정 | 추론 초점이 불명확 |
```

### 은쌤(사람 검수자)과의 관계
- 은쌤은 **교육과정 전문성** 기반의 최종 검수자
- Codex는 **구조적/체계적** 검수 담당
- 검수 순서: Claude 초안 → Codex 구조 검수 → 은쌤 교육과정 검수 → 승인

## 15. 자동 검증

### 검증 스크립트
- 위치: `scripts/validate.js`
- 실행: `node scripts/validate.js` (전체) 또는 `node scripts/validate.js --level 3` (특정 단계)

### 검증 항목
1. JSON 파싱 가능 여부
2. 필수 필드 존재 여부
3. ID 형식 (지문: `P-{단계}-{번호}`, 문항: `R-{단계}-{영역}-{번호}`)
4. passage_id 참조 무결성
5. 영역 코드 유효성
6. 성취기준 코드 형식
7. 단계(level) 범위 (1~7)
8. 선택지 수 (선택형 4개)
9. 정답 인덱스 범위
10. review_status 유효값
11. 중복 ID 검사
12. char_count 정확성

### 실행 시점
- Claude: 문항 초안 완성 후 반드시 실행
- Codex: 검수 전 실행하여 기본 구조 확인
- PR 생성 전 통과 필수

## 16. 웹 서비스 역할 분담 (Phase 3)

### 아키텍처
- 정적 HTML + JavaScript (서버리스)
- 문항 데이터는 JSON 파일에서 로드
- 결과는 클라이언트 사이드에서 계산
- 별도 백엔드, 데이터베이스, 서버 API는 Phase 3 범위에 포함하지 않는다.

### 역할 분담

| 영역 | Codex | Claude |

| 전체 아키텍처 | 설계/리뷰 | 구현 |

## 18. 최종 원칙

이 프로젝트에서 가장 중요한 것은 많이 만드는 것이 아니라,
나중에 다시 검수할 수 있는 구조로 정확하게 쌓는 것이다.

모든 초안은 수정 가능해야 하고,
모든 문항은 왜 그 단계인지 설명 가능해야 하며,
모든 판정은 데이터 구조와 평가 철학 위에서 일관되게 작동해야 한다.
