## 톤

### 기본 원칙
- **친근하되 전문적**: 동네 언니/오빠가 자기 일을 설명하는 느낌. 가볍지만 얕지 않게.
- **투명하고 솔직**: 실패도 공유. "처음엔 이렇게 했는데 안 됐어요"도 좋은 소재.
- **교육에 대한 진심**: 왜 이 일을 하는지, 어떤 가치를 추구하는지가 드러나야.

### 피해야 할 것
- 너무 딱딱한 보고서 어조 ("~하였음", "~으로 판단됨")
- 과도한 전문 용어 (불가피할 땐 괄호로 쉬운 설명 추가)
- 과도한 이모지 사용 (포인트에만 1~2개)
- 광고성 과장 표현

### 존대 기준
- 블로그 글: 존대말 기본 ("~입니다", "~했어요")
- 캐릭터 소개글: 약간 캐주얼 허용 ("~거든요", "~인데요")

## 글 구조

### 개발일지 (주 1회 정기)
```
제목: [개발일지 #N] 이번 주 질문연구소는요
날짜: YYYY-MM-DD

1. 인사 (1~2줄)
2. 이번 주 하이라이트 (핵심 1~2개)
3. 세부 작업 내용 (번호 목록, 각각 2~3줄)
4. 왜 이런 결정을 했는지 (설계 배경, 고민 과정)
5. 다음 주 예고
6. 마무리 인사
```

### 설립기 / 설계 철학 (비정기)
```
제목: 자유 형식
날짜: YYYY-MM-DD

1. 스토리 도입 (독자를 끌어들이는 질문이나 상황)
2. 배경과 맥락 (왜 이런 고민이 시작됐는지)
3. 결정 과정 (어떤 선택지가 있었고, 왜 이걸 골랐는지)
4. 결과와 의미 (이 결정이 어떤 영향을 미치는지)
5. 독자에게 질문 (참여 유도)
```

### 캐릭터 이야기 (비정기)
```
제목: [캐릭터 소개] OO을/를 만나보세요!
날짜: YYYY-MM-DD

1. 캐릭터의 한마디 (인용문 형태)
2. 누구인가요? (역할, 성격, 비주얼)
3. 탄생 비화 (왜 이 동물? 왜 이 성격?)
4. 질문연구소에서 하는 일
5. 재미있는 비하인드
```

### 교육 인사이트 (비정기)
```
제목: 자유 형식
날짜: YYYY-MM-DD

1. 문제 제기 (독자가 공감할 수 있는 상황)
2. 질문연구소의 시각
3. 근거와 분석 (교육과정, 연구, 데이터 등)
4. 실천 제안 (독자가 해볼 수 있는 것)
```

## 소재 수집 원칙

필로가 글을 쓸 때 참고하는 소재원:

| 소재원 | 용도 |
|--------|------|
| `git log` | 최근 커밋으로 무엇을 만들었는지 파악 |
| `memory/` 파일들 | 프로젝트 결정사항, 설계 배경 |
| 작업 산출물 (별쌤작업/ 등) | 구체적인 내용과 맥락 |
| `AGENT_ARCHITECTURE.md` | 시스템 설계 철학, 역할 분담 |
| 캐릭터 설정 (memory/project_characters.md) | 캐릭터 비화 |

## 파일 명명 규칙

```
blog/drafts/YYYY-MM-DD-{카테고리}-{슬러그}.md
blog/published/YYYY-MM-DD-{카테고리}-{슬러그}.md
```

카테고리 접두어:
- `devlog` — 개발일지
- `origin` — 설립기
- `philosophy` — 설계 철학
- `character` — 캐릭터 이야기
- `insight` — 교육 인사이트

예시:
- `2026-04-15-devlog-reading-items-complete.md`
- `2026-04-15-character-chaeksaem.md`
- `2026-04-15-origin-why-we-started.md`
