# 진단평가 엔진 설계 계획

## 목적
- 학생의 수준을 모르는 상태에서 진단을 시작한다.
- 단계별 문제은행을 활용해 올바른 수준으로 유도한다.
- 이전 단계 문제를 충분히 맞추면 다음 단계로 이동한다.
- 동시에 각 단계 안에서 상중하 변별력을 확보한다.

## 핵심 요구
1. 모든 단계별 문제는 미리 준비한다.
2. 첫 진단은 낮은 단계부터 시작한다.
3. 단계 상승/유지/종료 기준을 명확히 한다.
4. 진단 결과는 최종 수준과 추천 문제 출발 단계를 포함한다.

## 기본 로직 제안

### 1) 시작 단계
- 기본 시작 단계: 1단계
- 또는 간단한 스크리닝 문제 1~2개로 시작 단계 보정

### 2) 단계 내 흐름
- 각 단계마다 `n`문제씩 출제
- `n = 5` 이상 권장, 현재는 5문항 기준
- 각 단계에서 `3문제 이상 정답`이면 다음 단계로 이동
- `2문제 이하 정답`이면 그 단계 수준으로 판정
- `3문제 연속 정답` 방식도 사용 가능

### 3) 단계 간 이동 기준
- `1단계 → 2단계`: 1단계에서 3/5 이상 정답
- `2단계 → 3단계`: 2단계에서 3/5 이상 정답
- 하위 단계로 강제 하강은 초기에는 적용하지 않음

### 4) 변별력 처리
- 각 단계 안에서도 문제 난이도를 구분
  - 쉬움: 상위 단계 진입 조건을 확인하는 문제
  - 보통: 단계 중심 문제
  - 어려움: 단계 상
- 단계 상 문제는 다음 단계 진입 여부를 더 잘 구분
- 단계 하 문제는 현재 단계 판정을 안정화

## 데이터 구조 제안

### item 형식
- `item_id`
- `level` (1~7)
- `domain_code`
- `question_text`
- `choices`
- `answer`
- `passage_id`
- `difficulty_tier` (easy / medium / hard)
- `review_status`

### passage 형식
- `passage_id`
- `level`
- `text`
- `genre`
- `topic`
- `intended_domains`

### 진단 세션 상태
- `current_level`
- `current_domain` (필요 시)
- `attempted_items`
- `correct_count`
- `wrong_count`
- `history` (문제별 정답 여부)
- `next_level_candidate`

## 단계별 개발 순서
1. 현재 1~2단계 문제와 지문 데이터를 정리
2. `difficulty_tier` 메타를 추가
3. 단일 단계 문제은행 렌더링 컴포넌트 제작
4. 진단 흐름을 제어하는 상태 머신 설계
5. 사용자 흐름: 시작 → 문제 풀이 → 결과
6. 시뮬레이션 / 교사용 피드백 모드 추가
