# Codex 작업 규칙

> 이 파일은 Codex가 이 프로젝트에서 작업할 때 반드시 지켜야 하는 규칙이다.
> 모든 규칙은 예외 없이 적용된다.

---

## 1. 파일 보호 — 절대 규칙

### 1.1 삭제 금지

**어떤 파일도 삭제하지 않는다.**

- `rm`, `git rm`, 파일 내용을 빈 문자열로 덮어쓰기 모두 금지.
- "정리", "리팩토링", "구조 개선" 명목으로도 삭제하지 않는다.
- 파일이 불필요하다고 판단되면 삭제하지 말고 리포트에 기록한다.

### 1.2 전체 다시쓰기 금지

**기존 파일을 통째로 다시 쓰지 않는다.**

- 파일 전체를 새 내용으로 교체하는 행위 금지.
- 반드시 부분 수정(Edit)만 사용한다.
- 수정 범위가 파일의 50%를 넘으면, 수정하지 말고 리포트에 "대규모 수정 필요" 사안으로 기록한다.

### 1.3 잠금 파일 목록

아래 파일은 **읽기만 가능**하다. 어떤 상황에서도 수정하지 않는다.

```
# 아키텍처·설계 문서
AGENT_ARCHITECTURE.md
DATA_DESIGN.md
CLAUDE.md
AGENTS.md                    ← 이 파일 자체

# 에이전트 시스템 프롬프트
agents/philo.md
agents/byeolsaem.md
agents/chaeksaem.md
agents/geulsaem.md
agents/eunsaem.md
agents/hogi.md

# 교육과정 원문 데이터
reference/achievement-standards.json
reference/writing-achievement-standards.json
knowledge/curriculum/                      # 전체 디렉토리

# 준거틀
framework.json

별쌤-평가준거틀.md
글쓰기-평가준거틀.md
diagnosis/frameworks/                      # 전체 디렉토리

# 스키마
reference/schema.json
knowledge/schema/                          # 전체 디렉토리
```

### 1.4 제한적 수정 가능 파일

아래 파일은 **지정된 필드만** 수정할 수 있다.

| 파일 | 수정 가능 범위 | 금지 범위 |
|------|--------------|----------|
| `item-bank/**/items.json` | `review_status` 필드만 (`draft` → `reviewed`) | 문항 내용, 구조, ID, 정답 등 일체 |
| `item-bank/**/passages.json` | `review_status` 필드만 | 지문 텍스트, 메타데이터 등 일체 |
| `diagnosis/item-bank/**/items.json` | `review_status` 필드만 | 위와 동일 |
| `diagnosis/item-bank/**/passages.json` | `review_status` 필드만 | 위와 동일 |
| `COLLABORATION.md` | 검수 순서 관련 섹션만 | 역할 분담, 파일 소유권 섹션 |

## 2. 검수 모드 규칙

Codex가 이 프로젝트에서 검수 작업을 할 때는 아래 규칙을 따른다.

### 2.1 검수의 정의

**검수 = 읽기 + 판단 + 리포트 작성**

검수는 원본 산출물을 수정하는 행위가 아니다. 수정이 필요하면 리포트에 기록하고, 수정은 원래 생산자(Claude 또는 해당 에이전트)가 실행한다.

### 2.2 검수 산출물 위치

검수 결과는 반드시 아래 위치에만 쓴다:

```
review/reports/{날짜}-{대상}.md          # 검수 리포트
item-bank/**/검수-리포트.md             # 기존 구조 호환
diagnosis/item-bank/**/검수-리포트.md   # 새 구조 호환
```

기존 파일 안에 코멘트, 주석, 태그를 삽입하지 않는다.

### 2.3 검수 리포트 형식

```markdown
## 검수 리포트

- 대상: {파일 경로}
- 검수일: {날짜}
- 검수자: Codex
- 판정: PASS / CONDITIONAL PASS / FAIL

### 요약
{한 문장 요약}

### 항목별 결과
| 항목 | 판정 | 비고 |
|------|------|------|
| ... | PASS/FAIL | ... |

### 수정 필요 사항
| 대상 ID | 수정 유형 | 수정 내용 | 근거 |
|---------|----------|----------|------|
| ... | ... | ... | ... |

### 참고 의견
- {선택 사항}
```

### 2.4 검수 시 금지 행동

1. 원본 파일에 직접 수정을 가하지 않는다.
2. 파일을 삭제하거나 이름을 변경하지 않는다.
3. 새 파일을 만들 때 기존 파일과 같은 이름을 사용하지 않는다.
4. 검수 대상이 아닌 파일을 수정하지 않는다.
5. `review_status`를 `approved`로 변경하지 않는다. (`reviewed`까지만 가능, `approved`는 사람만)

## 3. 새 파일 생성 규칙

### 3.1 허용되는 새 파일 생성

- `review/reports/` 안의 검수 리포트
- `review/checklists/` 안의 체크리스트
- 검수 대상 폴더 안의 `검수-리포트.md`

### 3.2 금지되는 새 파일 생성

- 기존 파일의 "개선 버전", "v2", "수정본" 등의 이름으로 새 파일 만들기
- 기존 파일을 다른 이름으로 복사한 뒤 원본 삭제하기
- 아키텍처/설계 문서를 새로 작성하기 (기존 문서를 읽고 리포트에 의견 기록)

## 4. Git 규칙

### 4.1 브랜치

- 검수 작업은 `codex/review-*` 브랜치에서 한다.
- `main` 브랜치에 직접 커밋하지 않는다.
- force push 하지 않는다.

### 4.2 커밋

- 커밋 메시지에 수정한 파일과 사유를 명시한다.
- 예: `review: level-1 writing 문항 검수 리포트 작성`
- 예: `fix: R-1-VOC-003 review_status를 reviewed로 변경`

### 4.3 작업 전 확인

작업 시작 전 반드시:
1. `git status`로 현재 상태 확인
2. 수정되지 않은 clean 상태인지 확인
3. 작업 대상 파일 목록을 먼저 나열

## 5. 작업 보고 규칙

모든 작업이 끝나면 아래 형식으로 보고한다:

```markdown
## 작업 완료 보고

### 읽은 파일 (수정 안 함)
- {파일 경로}: {읽은 이유}

### 생성한 파일
- {파일 경로}: {내용 요약}

### 수정한 파일
- {파일 경로}: {수정 내용} (필드 변경만)

### 삭제한 파일
- (없음 — 삭제는 금지)

### 발견 사항
- {리포트에 기록한 핵심 내용}
```
