# 1단계 Claude API 테스트 세트

## 목적

이 문서는 **1단계 AI 피드백 프롬프트가 실제 Claude API에서 안정적으로 작동하는지 검증하기 위한 테스트 세트**입니다.

테스트 목표:

1. 합격/불합격 판정이 일관적인가
2. 실패 기준을 정확히 짚는가
3. 수정 예시가 실제로 도움이 되는가
4. 아직 배우지 않은 기준을 과잉 평가하지 않는가
5. 웹서비스에서 바로 렌더링할 수 있을 만큼 출력 형식이 안정적인가

---

## 사용 원칙

- 최소 6개 케이스를 모두 돌린다
- 가능하면 **같은 입력으로 2~3회 반복 호출**해 판정 흔들림을 확인한다
- 모델 출력은 반드시 JSON으로 받는다
- 프롬프트가 길어지더라도, 채점 기준은 해당 차시에 필요한 것만 포함한다
- 테스트 결과는 "판정 정확도"와 "피드백 품질"을 따로 본다

---

## 권장 테스트 순서

1. 차시 01, 04, 08, 12부터 먼저 돌린다
2. 그다음 차시 05, 07 같은 경계 판정이 많은 과제를 돌린다
3. 흔들리는 기준이 발견되면 공통 프롬프트를 먼저 수정한다
4. 그 후에 2단계 프롬프트 작성으로 넘어간다

---

## 테스트 케이스 구성

이 세트는 다음 4종류를 모두 포함합니다.

- **명확한 합격**: 대부분의 모델이 안정적으로 합격시켜야 하는 글
- **아슬아슬한 합격**: 조건은 충족하지만 표현이 다소 서툰 글
- **명확한 불합격**: 핵심 기준이 빠진 글
- **엣지 케이스**: 모델이 흔들리기 쉬운 글

---

## 케이스 1. 차시 01 명확한 합격

### 테스트 목적

- 주어/서술어/마침표 체크가 정확한지 확인
- 불필요한 맞춤법 평가를 하지 않는지 확인

### 입력 정보

- 차시: 01
- 과제: 오늘 있었던 일 3가지를 각각 완전한 문장 1개로 쓰기

### 학생 글

```
나는 아침에 학교에 갔다.
나는 쉬는 시간에 친구와 공놀이를 했다.
나는 저녁에 엄마와 책을 읽었다.
```

### 기대 판정

- 결과: 합격
- 핵심 이유:
  - 3문장 모두 주어 명시
  - 3문장 모두 서술어 있음
  - 3문장 모두 마침표 있음

### 체크 포인트

- "친구와"를 육하원칙으로 확장하라는 피드백이 나오면 안 됨
- 맞춤법/어휘 다양성 평가가 과하게 나오면 안 됨

---

## 케이스 2. 차시 01 명확한 불합격

### 테스트 목적

- 주어 생략을 정확히 잡는지 확인
- 문장별 위치 지목이 되는지 확인

### 입력 정보

- 차시: 01
- 과제: 오늘 있었던 일 3가지를 각각 완전한 문장 1개로 쓰기

### 학생 글

```
아침에 학교에 갔다.
나는 쉬는 시간에 친구와 공놀이를 했다.
저녁에 책을 읽었다.
```

### 기대 판정

- 결과: 아직 조금 더 다듬어봐요
- 핵심 이유:
  - 첫 번째 문장 주어 없음
  - 세 번째 문장 주어 없음

### 기대 수정 예시 방향

- "나는 아침에 학교에 갔다."
- "나는 저녁에 책을 읽었다."

### 체크 포인트

- 한국어에서 자연스러운 생략이라며 합격시키면 안 됨
- "전체적으로 조금 아쉬워요"처럼 모호하게 끝나면 안 됨

---

## 케이스 3. 차시 04 아슬아슬한 합격

### 테스트 목적

- 오류 개수 기준을 정확히 세는지 확인
- 학생을 지나치게 불합격시키지 않는지 확인

### 입력 정보

- 차시: 04
- 과제: 어제 한 일을 4문장으로 쓰기

### 학생 글

```
나는 어제 학교에서 발표를 했다.
나는 발표 전에 조금 긴장했다.
나는 집에 와서 숙제를 안 했다.
나는 오랜만에 일찍 잠을 잤다.
```

### 기대 판정

- 결과: 합격
- 핵심 이유:
  - 지정된 오류 항목 없음
  - 문장 수 충족

### 체크 포인트

- "안 했다"를 틀렸다고 하면 안 됨
- 차시 목표 밖의 감정 표현 빈약함을 문제 삼으면 안 됨

---

## 케이스 4. 차시 04 명확한 불합격

### 테스트 목적

- 지정된 맞춤법 항목을 안정적으로 감지하는지 확인
- 오류마다 개별 설명을 주는지 확인

### 입력 정보

- 차시: 04
- 과제: 어제 한 일을 4문장으로 쓰기

### 학생 글

```
나는 어제 발표를 했는데 너무 떨렸다.
나는 몇일 동안 연습한 내용을 말했다.
나는 집에 와서 숙제를 안했다.
나는 오랫만에 푹 쉬었다.
```

### 기대 판정

- 결과: 아직 조금 더 다듬어봐요
- 핵심 이유:
  - "몇일" 오류
  - "안했다" 띄어쓰기 오류
  - "오랫만에" 오류

### 기대 수정 예시 방향

- "나는 며칠 동안 연습한 내용을 말했다."
- "나는 집에 와서 숙제를 안 했다."
- "나는 오랜만에 푹 쉬었다."

### 체크 포인트

- 첫 문장까지 억지로 오류를 찾으면 안 됨
- 오류 개수 3개를 정확히 세는지 확인

---

## 케이스 5. 차시 08 엣지 케이스

### 테스트 목적

- 1단계 종합 과제에서 "구체성"과 "감정 간접 표현"을 구분하는지 확인
- 직접 표현이 있어도 간접 표현이 1개 있으면 통과 가능한지 확인

### 입력 정보

- 차시: 08
- 과제: 자유 주제 일기 8문장 이상

### 학생 글

```
오늘은 비가 많이 왔다.
나는 아침에 우산을 쓰고 학교에 갔다.
나는 교실에 들어가서 젖은 신발을 털었다.
나는 2교시에 발표를 했다.
발표를 시작할 때 내 손이 차가워지고 목소리가 작아졌다.
그래도 발표를 마치고 나니 기분이 좋았다.
나는 점심시간에 친구와 떡볶이를 먹었다.
오늘은 긴장됐지만 뿌듯한 하루였다.
```

### 기대 판정

- 결과: 합격 가능성이 높음
- 핵심 이유:
  - 8문장 충족
  - 사건 구체성 있음
  - 감정 간접 표현 1개 있음
  - 직접 표현도 있으나 전부 직접 표현만 있는 것은 아님

### 체크 포인트

- "기분이 좋았다"가 있다는 이유만으로 불합격시키면 안 됨
- 단, 주어 누락을 엄격하게 적용하면 일부 문장에서 흔들릴 수 있음

### 메모

이 케이스는 **실제 서비스에서 판정 편차를 보기 위한 경계 사례**입니다.
반복 호출 시 결과가 흔들리면, 차시 08 프롬프트에 아래 보정 문장을 추가하는 것이 좋습니다.

> 직접 감정 표현이 일부 있어도, 몸의 반응이나 행동으로 드러난 감정 표현이 1개 이상 있으면 해당 기준은 충족으로 본다.

## 케이스 6. 차시 12 명확한 불합격

### 테스트 목적

- 루브릭 종합 평가에서 점수와 코멘트가 연결되는지 확인
- 14점 미만일 때 합격시키지 않는지 확인

### 입력 정보

- 차시: 12
- 과제: 자유 주제 일기 8문장 이상

### 학생 글

```
오늘은 재미있었다.
나는 학교에 갔다.
나는 수업을 들었다.
나는 친구를 만났다.
나는 밥을 먹었다.
나는 집에 왔다.
나는 숙제를 했다.
나는 잤다.
```

### 기대 판정

- 결과: 아직 조금 더 다듬어봐요
- 예상 점수:
  - 유창성: 3
  - 구조: 2
  - 내용: 1
  - 어휘/표현: 1
  - 관례: 4
  - 총점 예시: 11점

### 기대 피드백 방향

- 칭찬: 문장 수를 채운 점
- 개선: 사건이 너무 단순 나열임
- 개선: 육하원칙과 감정 표현을 더 구체화할 것
- 수정 힌트: 한 장면을 골라 자세히 쓰기

### 체크 포인트

- 문장 수만 채웠다고 합격시키면 안 됨
- 관례 점수가 높아도 전체 합격은 아니어야 함

---

## 케이스 7. 차시 07 명확한 불합격 — 직접 표현 과다

### 테스트 목적

- 직접 표현이 3회 이상일 때 정확히 불합격시키는지 확인
- 간접 표현이 없을 때 C2 기준 실패를 잡아내는지 확인

### 입력 정보

- 차시: 07
- 과제: 오늘 일기에서 감정이 드러나는 문장을 3개 이상 포함해 일기 쓰기

### 학생 글

```
오늘은 학교에서 발표를 했다.
나는 발표가 끝나서 기뻤다.
나는 점심시간에 친구와 놀아서 좋았다.
집에 와서 엄마가 간식을 줬는데 신났다.
나는 오늘 하루가 즐거웠다.
나는 내일도 학교에 가고 싶다.
```

### 기대 판정

- 결과: 아직 조금 더 다듬어봐요
- 핵심 이유:
  - C2 실패: 간접 표현(몸의 반응) 없음
  - C3 위반: "기뻤다", "좋았다", "신났다", "즐거웠다" — 직접 표현 4회

### 기대 수정 예시 방향

- "나는 발표가 끝나고 나니 어깨에 힘이 빠지면서 저절로 웃음이 났다."
- "기뻤다/신났다 → 구체적인 몸의 반응으로 바꾸기"

### 체크 포인트

- "기뻤다", "좋았다"를 나열만 하고 합격시키면 안 됨
- "나는"이 있는 문장은 주어 있으므로 C1은 충족 — C2/C3만 실패여야 함
- 직접 표현 개수를 정확히 세는지 확인 (4개)

## 케이스 8. teacher_handoff 케이스 — 판단 불가 입력

### 테스트 목적

- 입력 글이 의미 없는 내용이거나 너무 짧을 때 teacher_handoff_needed=true로 처리하는지 확인
- 억지로 피드백하지 않는지 확인

### 입력 정보

- 차시: 01
- 과제: 오늘 있었던 일 3가지를 각각 완전한 문장 1개로 쓰기

### 학생 글

```
ㅎㅎ
모르겠어요
ㅋ
```

### 기대 판정

- 결과: teacher_handoff_needed=true
- teacher_handoff_reason: "제출한 내용이 과제 형식에 맞지 않아 AI가 평가할 수 없습니다."

### 체크 포인트

- pass=false이면서 teacher_handoff_needed=true여야 함
- improvements나 strengths에서 억지 피드백이 나오면 안 됨
- "더 열심히 써봐요" 같은 모호한 one_tip만 나오면 안 됨

## 먼저 수정해야 하는 흔들림 신호

다음 중 하나라도 나오면 2단계로 넘어가기 전에 1단계 프롬프트를 먼저 보정하는 것이 좋습니다.

- 같은 입력인데 합격/불합격이 반복 호출마다 바뀜
- 해당 차시에서 배우지 않은 요소를 지적함
- 실패한 기준 번호와 설명이 맞지 않음
- 수정 예시가 원문보다 더 어려워짐
- 초등 3학년 수준보다 어려운 어휘가 많이 나옴
- JSON 구조가 자주 깨짐
